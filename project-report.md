# Отчет о проделанной работе по проекту SourceMate

## 1. Цель работы

Целью работы было научиться выполнять верстку веб-сайта по заранее подготовленному макету в Figma, а также довести учебный проект до состояния работающего веб-приложения. Сайт должен соответствовать визуальной концепции макета, быть адаптивным, доступным, валидным с точки зрения HTML/CSS-структуры и включать полноценную пользовательскую логику.

В качестве проекта был разработан SourceMate - учебный веб-сервис для поиска научных источников, проверки учебных работ на возможные заимствования и формирования отчета по результатам проверки. Проект включает фронтенд-часть, серверную часть, API, работу с файлами, авторизацию пользователей, личный кабинет и дополнительные страницы.

## 2. Общая характеристика проекта

SourceMate предназначен для студентов, преподавателей и исследователей. Пользователь может зарегистрироваться, войти в личный кабинет, загрузить документ, указать тему работы и получить отчет с процентом оригинальности, найденными совпадениями и рекомендованными источниками.

В проекте реализованы:

- главная страница сайта;
- страница входа, регистрации и восстановления доступа;
- личный кабинет пользователя;
- загрузка документов форматов TXT, DOCX и PDF;
- проверка документа на совпадения с найденными источниками;
- детальная страница отчета;
- карточка научного источника;
- страница подписки;
- модальные окна настроек;
- виджет поддержки;
- экспорт отчета в PDF;
- серверное API;
- хранение пользователей, сессий и проверок;
- Telegram-бот для поиска источников.

## 3. Использованные технологии

Для реализации проекта использовались следующие технологии:

- HTML5 - структура страниц;
- CSS3 - стилизация, адаптивная верстка, сетки, визуальные эффекты;
- JavaScript - клиентская логика и интерактивность;
- Node.js - серверная часть;
- HTTP-модуль Node.js - веб-сервер без тяжелого фреймворка;
- Telegraf - Telegram-бот;
- formidable - обработка загрузки файлов;
- mammoth - извлечение текста из DOCX;
- pdf-parse - извлечение текста из PDF;
- PDFKit - генерация PDF-отчета;
- nodemailer и Brevo API - отправка кодов на почту;
- Ollama/локальная LLM - расширение запроса, оценка релевантности и рекомендации;
- Crossref, Semantic Scholar, OpenAlex, Wikipedia, CyberLeninka, НЭБ - источники научных данных.

## 4. Работа над фронтендом

Основной задачей фронтенда была верстка сайта по макету, подготовленному в Figma. При разработке учитывались размеры блоков, композиция, визуальный стиль, цветовая схема, типографика, состояния элементов и поведение интерфейса на разных экранах.

Были сверстаны следующие страницы:

- `index.html` - главная страница SourceMate;
- `auth.html` - авторизация, регистрация и восстановление доступа;
- `cabinet.html` - личный кабинет;
- `report.html` - детальный отчет проверки;
- `source.html` - карточка источника;
- `subscription.html` - тарифы и подписка.

Главная страница содержит первый экран с брендом SourceMate, навигацию, CTA-кнопку, блок результата, карточки с демонстрацией возможностей, преимущества сервиса, roadmap и футер с контактами. Для визуального соответствия макету использованы фоновые SVG-элементы, градиенты, карточки, изображения устройств, декоративные свечения и единая темная цветовая схема.

В личном кабинете реализована более сложная интерфейсная структура: верхняя навигация, приветственный блок, статистические карточки, история проверок, зона загрузки файла, боковая панель профиля, быстрые действия, блок рекомендованных источников, модальные окна настроек, виджеты загрузки и экспорта. Отдельно проработаны страницы отчета и источника, чтобы пользователь мог последовательно переходить от проверки к подробной информации.

## 5. Верстка и визуальная реализация

Верстка выполнена с использованием семантических HTML-элементов: `header`, `main`, `section`, `article`, `aside`, `nav`, `footer`, `address`. Это делает структуру страниц понятной как для браузера, так и для вспомогательных технологий.

Для построения интерфейса применялись:

- CSS Grid - для сложных сеток карточек, статистики и двухколоночных блоков;
- Flexbox - для навигации, кнопок, форм, чипов и компактных групп элементов;
- CSS custom properties - для основных цветов, размеров и повторяющихся значений;
- `clamp()` - для адаптивной типографики на главной странице;
- `max-width`, `min()`, `calc()` - для ограничения ширины контейнеров;
- `aspect-ratio` - для корректного отображения изображений устройств;
- `border-radius`, `box-shadow`, `radial-gradient` - для передачи визуального стиля макета;
- псевдоэлементы `::before` и `::after` - для декоративных фоновых эффектов.

Особое внимание было уделено совпадению с макетом: сохранены крупные заголовки, темный фон, светлая навигационная панель, градиентный знак бренда, карточная структура, скругления, отступы, визуальные акценты и расположение основных блоков.

## 6. Адаптивность

Сайт адаптирован под разные размеры экранов. В CSS используются медиазапросы для основных контрольных точек:

- до 1100 px - перестройка интерфейса личного кабинета;
- до 980 px - адаптация главной страницы и страницы авторизации;
- до 680 px - мобильная версия кабинета, модальных окон, истории проверок, отчета и подписки;
- до 620 px - мобильная версия главной страницы;
- до 360 px и 320 px - дополнительная корректировка для очень узких экранов.

На мобильных устройствах навигация упрощается, крупные сетки становятся одноколоночными, карточки получают более компактные отступы, часть действий переносится в мобильные панели и бургер-меню. Это позволяет пользоваться сайтом как на ноутбуке, так и на смартфоне.

## 7. Доступность интерфейса

При верстке были учтены базовые требования доступности:

- указан язык документа `lang="ru"`;
- добавлены `meta viewport` и корректная кодировка UTF-8;
- у страниц есть уникальные `title` и `description`;
- интерактивные элементы реализованы через `button` и `a`;
- у кнопок закрытия и навигации добавлены `aria-label`;
- у модальных окон используются `role="dialog"` и `aria-modal="true"`;
- у навигационных областей и секций добавлены `aria-label` и `aria-labelledby`;
- декоративные изображения скрываются пустым `alt=""` или `aria-hidden="true"`;
- важные изображения имеют осмысленный `alt`;
- для элементов, открываемых с клавиатуры, используются `tabindex` и роли.

Также была сохранена логичная иерархия заголовков, что важно для навигации по странице и восприятия структуры сайта.

## 8. Клиентская логика

Клиентская часть написана на JavaScript без использования тяжелых фронтенд-фреймворков. Скрипты отвечают за:

- переключение форм авторизации, регистрации и восстановления доступа;
- отправку запросов к API;
- работу с сессией пользователя;
- загрузку файла и отображение выбранного документа;
- запуск проверки в фоне;
- вывод истории проверок;
- удаление проверки;
- открытие детального отчета;
- открытие карточки источника;
- экспорт отчета;
- редактирование профиля;
- загрузку аватара;
- смену пароля;
- сохранение настроек;
- работу модальных окон;
- бургер-меню;
- виджет поддержки.

Для обмена с сервером используется `fetch`. Данные отправляются в формате JSON или `FormData`, если требуется передать файл.

## 9. Работа над бэкендом

Серверная часть реализована на Node.js. В проекте создан собственный HTTP-сервер, который раздает статические файлы из папки `public` и обрабатывает API-запросы.

На сервере реализованы:

- регистрация пользователей;
- подтверждение регистрации кодом на почту;
- вход и выход из аккаунта;
- восстановление доступа;
- хранение сессии через HTTP-only cookie;
- обновление профиля пользователя;
- смена пароля;
- загрузка аватара;
- сохранение пользовательских настроек;
- удаление аккаунта;
- управление подпиской;
- чат поддержки;
- загрузка документа для проверки;
- сохранение результатов проверки;
- экспорт отчета в PDF.

Данные учебного MVP хранятся в локальном JSON-файле `data/users.json`. При первом запуске сервер создает демонстрационного пользователя. Для паролей используется PBKDF2-хеширование с солью, что безопаснее хранения паролей в открытом виде.

## 10. Поиск источников и проверка документов

Бэкенд включает отдельный модуль агрегации источников. По пользовательской теме система обращается к нескольким внешним источникам данных: Crossref, Semantic Scholar, OpenAlex, Wikipedia, CyberLeninka и НЭБ. После получения результатов выполняется дедупликация, языковая фильтрация, лексическая оценка релевантности и ранжирование.

Дополнительно используется локальная LLM. Она помогает расширять поисковый запрос, формировать профиль темы, оценивать релевантность источников и выдавать рекомендации по найденным совпадениям.

Проверка документа выполняется следующим образом:

1. Пользователь загружает TXT, DOCX или PDF.
2. Сервер извлекает текст из файла.
3. Определяется тема проверки или используется тема, введенная пользователем.
4. По теме выполняется поиск научных источников.
5. Система пытается загрузить тексты найденных источников.
6. Документ сравнивается с источниками по шинглам.
7. Рассчитываются проценты оригинальности и совпадений.
8. Формируется список похожих источников и проблемных фрагментов.
9. Пользователь получает отчет в личном кабинете.

## 11. Оптимизация контента

В ходе работы была проведена оптимизация контента и структуры страниц:

- изображения вынесены в папку `public/assets`;
- SVG используется для логотипа, иконок и фоновых декоративных элементов;
- для страниц кабинета, отчета, источника и подписки добавлены `prefetch`-ссылки, чтобы ускорить переходы между связанными экранами;
- CSS разделен по назначению: главная страница, авторизация, кабинет и общие компоненты;
- повторяющиеся визуальные решения оформлены через классы и CSS-переменные;
- изображения имеют `max-width: 100%`, что предотвращает выход за пределы контейнера;
- для крупных визуальных блоков заданы стабильные размеры и пропорции;
- сервер ограничивает размер загружаемых документов до 25 МБ, а аватаров - до 5 МБ;
- сервер ограничивает количество источников, которые загружаются для проверки, чтобы не перегружать систему;
- в алгоритме поиска используется ограниченная параллельность и таймауты для внешних запросов и LLM.

Контент страниц был структурирован так, чтобы пользователь сразу понимал назначение сервиса, мог перейти к регистрации, загрузить работу и получить результат без лишних действий.

## 12. Валидность кода

HTML-код проекта построен на современном HTML5. Во всех основных страницах указаны `<!doctype html>`, язык документа, кодировка, viewport и метаописание. Разметка разделена на смысловые блоки, используются семантические теги и корректные типы элементов управления.

CSS-код вынесен в отдельные файлы. Для адаптивности используются медиазапросы, Grid, Flexbox и современные CSS-свойства. JavaScript-код разделен по страницам и отвечает за конкретные сценарии интерфейса.

Проверка валидности в рамках проекта предполагает:

- отсутствие незакрытых HTML-тегов;
- корректное подключение CSS и JavaScript;
- наличие `alt` у изображений;
- наличие подписей и доступных названий у интерактивных элементов;
- отсутствие критических ошибок в консоли браузера при основных сценариях;
- корректную обработку ошибок API;
- проверку работы форм и загрузки файлов;
- проверку адаптивности через инструменты разработчика.

## 13. Безопасность и обработка данных

В проекте реализованы базовые меры безопасности:

- пароли не хранятся в открытом виде;
- для паролей используется PBKDF2-хеш с солью;
- сессия хранится в HTTP-only cookie;
- cookie имеет флаг `SameSite=Lax`;
- коды подтверждения имеют ограниченный срок действия;
- загружаемые файлы ограничены по размеру;
- загрузка аватаров ограничена изображениями PNG, JPG, WebP и GIF;
- пользователь может выйти из аккаунта, сменить пароль и удалить аккаунт;
- сервер возвращает ошибки в JSON-формате без раскрытия лишних внутренних данных.

## 14. Итоги работы

В результате работы был создан адаптивный веб-сайт SourceMate, соответствующий задаче по верстке сайта по макету. Проект не ограничился статической версткой: была реализована полноценная клиентская и серверная логика, личный кабинет, авторизация, загрузка документов, проверка текста, формирование отчета и экспорт результата.

Работа позволила закрепить навыки:

- анализа макета в Figma;
- переноса визуальной структуры макета в HTML и CSS;
- построения адаптивной верстки;
- разработки интерактивных пользовательских сценариев;
- работы с формами и файлами;
- создания серверного API;
- интеграции фронтенда и бэкенда;
- оптимизации контента;
- проверки доступности и валидности кода.

Проект можно развивать дальше: подключить полноценную базу данных, добавить экспорт отчетов в другие форматы, расширить список источников, улучшить алгоритм проверки оригинальности, доработать платежную модель подписок и добавить административную панель.
